01 — Users & goals
Our end users are not technical. They are pharmacists and pharmacy managers who currently manage purchases on paper or in WhatsApp. Every screen must guide them step by step — no assumed knowledge, no jargon.

PH
Pharmacist (daily user)
Receives supplier deliveries, enters invoices, adds products to reorder list. Works fast, under pressure, often on mobile. Does not understand accounting terms like "debit note" or "COGS."
Pain: types the same invoice data 3x per week
MG
Pharmacy manager (weekly user)
Reviews what was purchased, follows up on unpaid invoices, approves returns, checks if supplier credits came back. Needs a clear dashboard view and payment status per invoice.
Pain: doesn't know what's owed to which supplier
02 — Master workflow
Every purchase interaction fits into one of three paths. The system must make the right path obvious from the first tap.

Path A
Create invoice from delivery
Path B
Reorder from wish list
Path C
Return goods to supplier
Path A — Enter a supplier delivery invoice
Supplier arrives → pharmacist opens "create invoice" → selects supplier → adds products one by one (or via OCR scan) → sets quantities, prices, expiry dates → saves draft → confirms when ready → inventory is updated automatically.

Path B — Reorder low-stock items
System auto-populates wish list from stock levels → pharmacist opens wish list → reviews items → taps "create purchase" → system pre-fills a new invoice with those items and the last supplier → pharmacist confirms quantities and prices → confirms.

Path C — Return goods to supplier
Pharmacist opens "create return" → selects supplier → enters original supplier invoice date + number → adds products to return with quantities → sets return price → confirms → inventory decremented → return register updated with payment status.

SCR-01
Step-by-step UX flow — pharmacist view
For UI/UX team
1
User opens "Create invoice"
PO number and timestamp auto-fill. All other fields blank. Confirm button is disabled (greyed out) until validation passes.
Show a blue hint banner: "Have the paper invoice? Tap the camera icon to fill this form automatically."
2
User selects supplier
Searchable dropdown. As soon as supplier is selected, show their last 3 invoices as a small history chip below the field — builds confidence.
If only one supplier exists, auto-select it and skip this step visually.
3
User fills invoice number and date
Both optional but encouraged. Show helper text: "Enter the number printed on the paper invoice." Date defaults to today.
 If OCR was used, these are already filled — skip to step 5.
4
User sets payment method and status
Payment method persists from last invoice (most pharmacies use one method per supplier). Payment status defaults to "Pending."
5
User searches for a product to add
Search by product name or barcode scan. Autocomplete dropdown shows: product name, current stock level, last purchase price from this supplier.
On mobile: show a large barcode scan button alongside the search field.
6
Product row appears — user fills details
Expiry date, quantity, free goods, purchase price, discount % are the key inputs per row. Sale price pre-fills from product master. Tax pre-fills from product category.
If expiry date is within 3 months: show orange warning "Expiry date is close — confirm with supplier."
7
Purchase price entered — AI anomaly check fires
System checks entered price against historical average for this SKU + supplier combination.
 If deviation >20%: show inline warning "This price is X% higher than your last purchase from [supplier]. Continue?"
8
User repeats step 5–7 for each product
Grand total updates live at the bottom after every product addition or edit.
9
User taps "Save draft" or "Confirm order"
Draft: saves without inventory update, no validation required except PO number. Confirm: runs full validation, then updates inventory and locks the invoice.
Block confirm if: no supplier selected, no products added, any product has qty = 0, total = 0.
10
Confirmation success screen
Show: PO number, supplier name, total value, product count. Two buttons: "View invoice" and "Create another." Do NOT auto-navigate away — user may need to print.
Invoice state machine:
Draft
→ confirm →
Received
→ pay →
Paid
Draft can be edited freely. Received is locked for editing — only payment status can change. Paid is fully locked. Cancellation from Draft or Received requires manager-level permission.

05 — Screen: wish list
Reference: Screenshot 3. This is the bridge between inventory management and purchasing. A pharmacist should open this screen when they need to reorder — and find it already populated with what needs buying.

SCR-03
Auto-population logic (AI-assisted)
1
System calculates reorder point per product
Reorder point = average daily sales × lead time days (default: 7 days). Recalculates nightly from sales data.
 Background job runs at 2am daily. No manual trigger needed.
2
Product appears in wish list when stock ≤ reorder point
System adds it with: product name, recommended quantity (= reorder qty − current stock), last supplier who provided this product.
3
Pharmacist can also manually add any product
Search field at top of wish list. Manually added items marked with a "Manual" badge so they're visually distinct from auto-added ones.
4
Pharmacist reviews list, edits quantities if needed, then taps "Create purchase"
System groups selected items by last supplier → creates one draft invoice per supplier → pharmacist is taken to the invoice creation screen with items pre-filled.
If items span 3 suppliers: show confirmation "This will create 3 draft invoices — one per supplier. Continue?"
5
After invoice is confirmed, remove items from wish list
Items linked to a confirmed invoice are automatically removed. Items linked to a draft invoice are marked "In draft PO-{n}" so they're not double-ordered.

08 — AI features specification
Three AI features are included. Each solves a specific manual burden. No chatbots, no conversational UI. AI runs silently in the background or triggers as an inline helper — never as a separate screen.

AI Feature 1: Invoice OCR from camera scan
1
What it does
Pharmacist taps the camera icon on the create invoice screen. Takes a photo of the paper supplier invoice. AI extracts: supplier name, invoice number, invoice date, product lines (name, quantity, price). Pre-fills the form. Pharmacist reviews and corrects anything before confirming.

Trigger: camera button on create invoice screen header
2
Dev implementation notes
Use a document OCR API (Google Vision, AWS Textract, or Azure Form Recognizer). Send image as base64. Parse response to extract table-structured invoice data. Map extracted product names to internal product catalog via fuzzy match (confidence threshold: 80%). Fields below threshold shown highlighted in yellow — user must confirm manually.

Show extraction confidence per field: green tick (≥80%), yellow warning (<80%), red x (not found)
3
UX behaviour
After scan: show loading spinner "Reading your invoice..." (max 5 seconds). On success: form pre-fills with a blue "AI filled" badge on each auto-filled field. On failure: "We couldn't read this invoice clearly — you can fill it in manually." Never block the user from proceeding manually.

AI Feature 2: Purchase price anomaly detection
1
What it does
When a pharmacist enters a purchase price for a product, the system compares it to the rolling average of the last 10 purchase prices for that product from the same supplier. If the new price deviates by more than 20%, show an inline warning — not a blocking error — so the pharmacist can verify before saving.

Trigger: on blur of purchase price field on any invoice line
2
Warning message copy
"This price (EGP X) is Y% higher than your average from [Supplier] (EGP Z). Double-check the paper invoice." — with a "Looks correct, continue" dismiss button. Do NOT block saving. Log all warnings and dismissals for manager reporting.

3
Data requirement
Requires at least 3 prior purchase records for this product+supplier combination to fire. If fewer than 3 records exist: do not show warning. Store price history per product per supplier in purchase_price_history table.

AI Feature 3: Automatic wish list population
1
What it does
A nightly background job scans every active product. For each product: calculates average daily sales (last 30 days) × 7 days lead time = reorder point. If current stock ≤ reorder point AND product is not already in wish list AND no pending/draft purchase order exists for this product: add to wish list automatically.

Trigger: cron job, runs at 2:00am daily
2
What gets pre-filled on wish list entry
Recommended order quantity = reorder point × 2 (two weeks supply) − current stock. Last supplier = most recent confirmed supplier for this product. Both are editable by pharmacist before creating the purchase.

3
Edge cases
Products with zero sales in last 30 days: skip (no demand signal). Products with no purchase history: skip. Products already on a draft PO: mark as "In draft — PO-{n}" instead of adding again.

All columns per SCR-02. Dashboard tiles (this month count, this month value, drafts count). Default sort newest first. Pagination (10/25/50 per page). Inline payment status toggle for manager role.

PUR-009
Wish list — manual add & display
P2
Wish list screen with manual product search and add. All columns per SCR-03 spec. Inline quantity edit. "Create purchase" button that groups by last supplier and creates draft invoices. Remove item action.

PUR-010
Wish list — auto-population cron job
P2
Background job at 2am daily. Calculate reorder point per product (avg daily sales × 7). If stock ≤ reorder point and not already in wish list and no pending PO: add to wish list with recommended qty and last supplier. Skip products with <3 sales records.

PUR-011
AI: Invoice OCR scan
P3
Camera button on create invoice screen. Integrates with OCR API. Parse response for: supplier name, invoice number, invoice date, product table. Fuzzy-match product names to catalog (80% threshold). Show confidence badges per field (green/yellow/red). Never block manual entry if OCR fails.

PUR-012
AI: Purchase price anomaly detection
P3
On blur of purchase price field: fetch rolling average of last 10 purchase prices for this product+supplier. If new price deviates >20%: show inline dismissable warning. Requires purchase_price_history table. Only fires if ≥3 historical records exist. Log all warnings and user dismissals.

PUR-013
PDF download — invoice & return printout
P2
Downloadable PDF for individual invoices and returns. Include: PO/RPO number, date, supplier details, product lines table, totals breakdown, payment status. Professional layout — pharmacies send these to their accountant.

PUR-014
Validation rules — all screens
P1
Implement all 11 rules from section 09. Block rules: prevent confirm and show toast. Warn rules: show inline dismissable warning, still allow proceed. All error messages use the exact copy from the spec.
